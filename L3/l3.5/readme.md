## Файлы для EventBooker — сервис бронирования с дедлайнами  

### 📋 Описание проекта  

Когда люди записываются на мероприятия (концерты, мастер-классы, курсы, лекции) — часто  
возникает ситуация: кто-то бронирует место, но не оплачивает вовремя. Организатору  
важно не держать такие «мертвые души», а освободить место для других. Поэтому в реальных  
системах бронирования всегда есть правило: если не оплатил (подтвердил) в течение  
определённого времени — бронь аннулируется.  

В этом проекте реализован сервис, который позволяет:  
- создавать мероприятия;
- бронировать места;  
- подтверждать (оплачивать) бронь;  
- автоматически отменять неоплаченные брони через заданный интервал времени.  

Бронирование не «вечное»: для каждого мероприятия можно задать свой срок жизни брони (в минутах).  
Если в течение этого времени бронь не подтверждена — она отменяется фоновым процессом, и место  
снова становится доступным.  

Вся логика реализована с использованием транзакций PostgreSQL для гарантии целостности данных,  
а для отслеживания и обработки просрочек задействованы Redis (ZSET для хранения времени истечения)  
и RabbitMQ (очередь сообщений об отмене).  
Такой подход обеспечивает надёжность и масштабируемость фоновой обработки.  

### 🖥️ Возможности  

Встроенные HTTP-методы:  

  - **POST /events** — создание мероприятия (название, дата, количество мест, время жизни брони, стоимость);  
  - **POST /events/{id}/book** — бронирование места (пользователь привязывается к брони);  
  - **POST /events/{id}/confirm** — подтверждение (оплата) брони;  
  - **GET /events/{id}** — получение информации о мероприятии и количестве свободных мест;  
  - **GET /events** — список всех предстоящих мероприятий;  
  - **POST /register** — регистрация нового пользователя;  
  - **POST /login** — вход (получение ID пользователя по email);  
  - **GET /events/{id}/book?user_id=...** — проверка, есть ли у пользователя бронь на мероприятие (возвращает ID брони и статус).  

Фоновый обработчик:  
- Каждую секунду проверяет в Redis просроченные брони (с помощью ZSET);  
- Найденные ID отправляются в RabbitMQ;  
- Воркер читает очередь и отменяет бронь в базе данных (освобождая место).  

Простой веб-интерфейс (HTML + CSS + JS):  
- Страница администратора (/admin.html) — создание мероприятий, просмотр списка;  
- Страница пользователя (/user.html) — регистрация/вход, просмотр мероприятий, бронирование и подтверждение;  
- Главная страница (/) — выбор роли.  

Дополнительно (в проекте реализовано):  
- регистрация нескольких пользователей и учёт их в бронированиях (один пользователь — одно место на мероприятии);  
- возможность настраивать разный срок жизни брони для каждого мероприятия (поле bookingTTLMinutes);  
- после подтверждения брони запись удаляется из Redis (больше не будет отменена).  

### 🗂️ Структура проекта  

```bash
├── cmd/
│   └── main.go             # точка входа (запуск всех компонентов)
├── pkg/
│   ├── api/                # обработчики HTTP и модели запросов/ответов
│   ├── broker/             # клиент RabbitMQ (публикация/потребление)
│   ├── configuration/      # загрузка конфигурации из .env
│   ├── db/                 # работа с PostgreSQL (модели, запросы, миграции)
│   ├── domain/             # общие доменные структуры
│   ├── overdueWorker/      # воркер, обрабатывающий сообщения из очереди
│   ├── server/             # запуск HTTP-сервера (Gin + middleware)
│   ├── service/            # бизнес-логика (проверки, вызовы БД и Redis)
│   └── zSet/               # клиент Redis для работы с ZSET (просрочки)
├── web/                    # статические файлы веб-интерфейса
│   ├── index.html
│   ├── admin.html
│   ├── user.html
│   ├── style.css
│   └── script.js
├── compose.yml             # оркестрация всех сервисов (app, postgres, redis, rabbitmq)
├── Dockerfile              # сборка образа приложения
├── .env                    # переменные окружения (пример)
├── go.mod, go.sum          # зависимости Go
└── readme.md               # этот файл
```

### 🚀 Быстрый старт  

**Требования:**  

- Docker  
- Свободные порты:  

    8081 — веб-интерфейс и API  
    5432 — PostgreSQL  
    6379 — Redis  
    5672 — RabbitMQ (AMQP)  
    15672 — веб-консоль RabbitMQ  

**Docker-образы**  

- postgres:16  
- redis:7  
- rabbitmq:4.0  

**Запуск системы:**  

    cd ./ && docker compose up  


- После успешного запуска веб-интерфейс будет доступен по адресу: http://localhost:8081  

### ⚙️ Конфигурация  

Все настройки задаются через файл .env в корне проекта.

    ## переменные сервера и api
    SERVICE_HOST_NAME=0.0.0.0          # хост, на котором слушает сервис
    SERVICE_PORT=8081                  # порт сервера
    GIN_MODE=debug                     # режим логов Gin (debug / release)

    ## переменные базы данных
    DB_HOST_NAME=postgres              # имя контейнера PostgreSQL в сети Docker
    DB_PORT=5432                       # порт PostgreSQL
    DB_NAME=postgres                   # имя базы данных
    DB_PASSWORD=postgres               # пароль
    DB_USER=postgres                   # пользователь

    ## переменные Redis (ZSET для просрочек)
    REDIS_HOST_NAME=redis              # имя контейнера Redis
    REDIS_PORT=6379                    # порт Redis
    REDIS_PASSWORD=                    # пароль (если есть)
    REDIS_DB=0                         # номер базы данных Redis
    ZSET_OVERDUE_KEY=bookings          # ключ ZSET для хранения просрочек
    ZSET_CHECK_INTERVAL=1s             # периодичность проверки просрочек
    REDIS_READ_RETRY_ATTEMPTS=3        # кол-во повторов при чтении
    REDIS_READ_RETRY_DELAY=100ms       # начальная задержка повторов
    REDIS_READ_RETRY_BACKOFF=2.0       # множитель задержки

    ## переменные брокера (RabbitMQ)
    RABBIT_HOST_NAME=rabbitMQ          # имя контейнера RabbitMQ
    RABBIT_PORT=5672                   # порт AMQP
    RABBIT_USER=rabbitMQ               # пользователь
    RABBIT_PASSWORD=rabbit             # пароль
    RABBIT_VHOST=/                     # virtual host
    RABBIT_QUEUE=bookingQueue          # имя очереди для просрочек
    RABBIT_EXCHANGE=bookingExchange    # exchange
    RABBIT_ROUTING_KEY=booking         # routing key
    RABBIT_CONNECTION_NAME=eventbooker # имя соединения
    RABBIT_CONNECT_TIMEOUT=5s          # таймаут подключения
    RABBIT_HEARTBEAT=10s               # heartbeat  
    RABBIT_RETRY_ATTEMPTS=3            # повторы публикации/потребления  
    RABBIT_RETRY_DELAY=3s              # начальная задержка  
    RABBIT_RETRY_BACKOFF=2.0           # множитель задержки
